# 0005. Unify field validation across nesting via a generated shared field-item schema

## Context

A field is validated differently depending on where it sits:

- **Top-level** (`acf.schema.json` → `fields[].items`): `unevaluatedProperties: false` over `allOf[ base `field.schema.json` + the per-type `if/then` branches ]`. Strict and type-aware.
- **Nested** (`refs/field-repeater.schema.json`, `refs/field-flexible_content.schema.json` → `sub_fields`): `items: { "$ref": "field.schema.json" }` — base only. No per-type branch, no `unevaluatedProperties`.
- **Group** (`refs/field-group.schema.json` → `sub_fields`): `type: array` only — not validated at all.

So the same field JSON can pass when nested yet fail at the top level, and junk/typos pass silently when nested. The visible symptom (issue #8): a top-level field whose per-type ref (`refs/field-<type>.schema.json`) is an **empty stub** has every type-specific key it legitimately emits flagged as "unevaluated" and rejected. Probing each empty stub against the bundled schema proved **10 affected types** (`file`, `oembed`, `user`, `page_link`, `relationship`, `clone`, `tab`, `time_picker`, `date_picker`, `date_time_picker`, `button_group`); `email`/`url`/`password` are correctly empty (they emit nothing beyond base).

The per-type `if/then` branches are generated by `SchemaEmitter` from `FIELD_TYPE_ORDER`. Hand-duplicating them into each `sub_fields` position would drift from that list — exactly what `SchemaConsistencyTest` exists to prevent.

## Decision

Introduce **one generated gate schema** and reference it from every field position.

- `SchemaEmitter` emits `schemas/field-item.schema.json` = `allOf[ {$ref: refs/field.schema.json} + the per-type if/then branches ]` with `unevaluatedProperties: false` — the construct that currently lives inline in `acf.schema.json`'s `fields[].items`.
- It is a **generated artifact** living in `schemas/` (next to `acf.schema.json`), *not* in `schemas/refs/`. It is therefore exempt from the byte-identical dual-copy rule of ADR [0002](0002-refs-byte-identical-dual-copy.md), which governs the hand-curated refs; a separate WP-free test pins it to the emitter's output instead.
- Every field position references it: `acf.schema.json` `fields[].items`, `field-repeater`/`field-flexible_content` `sub_fields.items`, and `field-group` `sub_fields.items` (previously unvalidated). Recursion (repeater-in-repeater) resolves through a self-referential `$ref`.
- The 10 incomplete per-type stubs are completed so the now-uniform strictness does not reject valid nested fields. Property sets are curated from **real ACF Pro exports harvested across 15 on-disk projects** (147 field instances), not from guesswork.

## Consequences

- A field validates identically regardless of nesting depth; the issue-#8 class is closed for every type, and `group` sub-fields gain per-item validation for the first time.
- Nested junk/typos are now rejected (intended tightening) — but an incomplete stub now also breaks valid *nested* fields, so completing all 10 stubs is a hard precondition that ships in the same change, not a follow-up.
- **Plugin-augmented exports fail.** Real exports from projects running WPGraphQL-ACF (`show_in_graphql`, `graphql_*`) or Relevanssi (`relevanssi_exclude`) carry keys this bundle deliberately does not model. They already fail at the top level today; unification extends that to nested fields. This is a known limitation, not a regression — the bundle targets core ACF Pro. Modeling plugin keys is out of scope.
- `clone` has no on-disk export sample; its stub is curated from ACF source and flagged for live-WP verification before release.
- **Guards that must stay true:** a new WP-free test pins `schemas/field-item.schema.json` to `SchemaEmitter` output; a new symmetry test asserts top-level and nested validation agree for a representative field of each type (the direct regression test for this asymmetry); the existing `FixtureValidationTest` real-export fixtures guard against over-tight constraints; `SnapshotTest` (WP-gated) still validates a full regeneration.
- The generated surface grows by one file (`field-item.schema.json`); regenerating via `bin/acf-schema-gen` against a live WP install remains the pre-release step.
